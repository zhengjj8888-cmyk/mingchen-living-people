# 明辰 Living People

一个把中文和英文表达润色得更自然、但仍然**像你自己在说**的 Codex Skill。

它适合口播、社交内容、文章草稿、产品说明和商务沟通。它不追求把所有文字改成同一种“网感文案”，而是先守住你的事实、观点和语气，再处理机器腔、空话、长句和不顺的节奏。

本仓库中的流程、检查标准和示例由明辰原创设计与维护，并以 [MIT License](LICENSE) 开源。

## 它能处理什么

| 你的需求 | 会得到什么 |
| --- | --- |
| “这段太像 AI 了” | 保留原意、更自然的改写稿 |
| “改成 1 分钟口播” | 短句、有停顿、能直接读的口播版 |
| “帮我看看哪里不对” | 2–3 个最影响阅读的问题，不擅自重写全文 |
| “压缩到 200 字” | 保住结论、事实和行动的精简稿 |
| “按我的语气写” | 学习你提供样稿的节奏与直接程度，不复制句子或杜撰经历 |
| “英文也顺一下” | 清楚、保真、适合实际沟通的英文润色 |

## 第一次使用：按这 4 步来

### 1. 安装到 Codex

在终端运行：

```bash
cd ~/Desktop
git clone https://github.com/zhengjj8888-cmyk/mingchen-living-people.git
mkdir -p ~/.codex/skills
cp -R mingchen-living-people ~/.codex/skills/
```

也可以下载 ZIP，解压后把 `mingchen-living-people` 文件夹完整复制到 `~/.codex/skills/`。

### 2. 重启 Codex

重启后，Codex 才会识别 `$mingchen-living-people`。

### 3. 粘贴原文，说明你想要什么

最简单的说法：

```text
使用 $mingchen-living-people 润色下面这段文字。
保留我的观点、数字和第一人称，不要写得像营销号。
直接给我改完的版本。

[粘贴原文]
```

### 4. 对照原文读一遍

确认专名、数字、时间、引用、立场没有被改掉；口播稿最好自己读一遍，检查是否顺口。该 Skill 的重点是“保留你”，不是“替你换一个人设”。

## 五个可以直接复制的提示词

### 轻润色：让文字顺一点

```text
使用 $mingchen-living-people 轻润色这段文字。
保留原来的结构、长度、事实和语气，只改生硬、重复和不顺的地方。
不要加案例、数据、经历或营销话术。直接给成稿。

[原文]
```

### 改成自然口播

```text
使用 $mingchen-living-people，把下面内容改成 60 秒中文口播。
要像我在认真讲一件亲身观察到的事：短一点、口语一点、有自然停顿。
不要硬广，不要“颠覆、赋能、风口”这类大词；没有的经历不要补。
直接给口播稿。

[原文]
```

### 只做表达诊断

```text
使用 $mingchen-living-people，只诊断这段文字为什么读起来像 AI。
请指出最影响表达的 3 个问题，每个问题给一句最小修改示例。
先不要重写全文。

[原文]
```

### 压缩到指定字数

```text
使用 $mingchen-living-people，把下面内容压缩到 180 字以内。
必须保留：核心结论、[这些数字/专名/行动]。
能删背景和重复，但不要把推测写成事实。
先给成稿；若有内容因字数无法保留，再用一句话说明。

[原文]
```

### 参考自己的样稿，但不复制

```text
使用 $mingchen-living-people。
下面第一段是我的写作样稿，请学习它的句长、语气和直接程度；
第二段是要改的内容。不要抄样稿里的独特表达，也不要补写样稿没有的经历。
直接输出改写后的第二段。

[样稿]
[要改的内容]
```

## 这个 Skill 的工作方法

```text
原文 + 目标平台 + 想保留的东西
             ↓
识别任务：诊断 / 轻润色 / 口播 / 压缩 / 重写
             ↓
锁定不可变内容：事实、数字、专名、引用、立场
             ↓
观察作者节奏：句长、用词、停顿、直接程度
             ↓
最小必要改写
             ↓
回读：还像原来的意思和这个人吗？
```

它会优先做这些调整：

- 删除没有信息量的开场、重复形容词和模板化号召。
- 拆开太长的句子，把抽象判断落回你已经提供的场景、动作或观察。
- 让口播有自然停顿，而不是书面连接词堆叠。
- 在正式表达中优先保证清楚、准确和边界感。
- 原文已经自然时，只修明显问题，不为了改而改。

## 一个简短示例

原句：

> 在人工智能快速发展的背景下，企业应当从战略高度重视内容生产流程的智能化升级。

口播式改写：

> AI 发展得很快，但企业最先该改的，往往不是口号，而是每天都在重复的内容流程。

这里改变的是表达顺序与节奏，不是替你新增观点、客户案例或转型成果。

## 什么不会做

- 不编造你的经历、情绪、客户、数据、来源或成果。
- 不把一句观察扩写成已经发生的事实。
- 不因为“去 AI 味”就删掉必要的术语、限定条件或证据。
- 不用夸张结论、虚假紧迫感或硬性号召换取“更有传播力”。
- 不复制参考样稿里的独特句子或作者标识。

如果某项事实需要替换，应该保留 `[待确认]`，而不是悄悄补一个看似合理的答案。

## 常见问题

**它会不会把我写得很像“同一个 AI”？**  
不会。它先看你的原文或样稿，优先保留你已有的用词、节奏和立场。没有样稿时，也只做最小必要调整。

**我只想知道问题，不想被全文重写。**  
在提示中写“只诊断，不重写全文”。Skill 会只给最关键的问题和局部示例。

**口播和正式邮件能用同一个方式吗？**  
不建议。口播要顺口、有停顿；正式沟通要准确、可执行。请在需求里说明使用场景。

**只有一句话也能用吗？**  
可以。短文本更应该少改，避免为了“自然”而无意义扩写。
